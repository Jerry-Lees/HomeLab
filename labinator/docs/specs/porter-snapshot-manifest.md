# Porter Snapshot Manifest — Specification Notes

These manifests are generated by **porter** (`~/projects/porter`) — the dual-pane terminal
file manager built for homelabs and sysadmin work. Porter produces them as part of its
system snapshot → diff → archive workflow. When working on labinator's archive ingestion,
refer back here for the exact schema.

---

## How Manifests Are Created

1. User navigates to a directory (or `/`) in a porter pane
2. Right-click background → **Take Snapshot** — porter walks the tree and records
   `(mtime, size, mode, uid, gid)` for every non-hidden file
3. User modifies files on the system
4. Right-click background → **Build Archive from Diff** — porter re-walks, compares
   against the snapshot, lists changed/new files in a dialog
5. User confirms and names the archive → porter builds a `.tar.gz` (or `.zip`) containing
   the changed files plus a `manifest.yaml` at the archive root

The manifest file inside the archive is always named **`manifest.yaml`**.

---

## Archive Layout (example)

```
<archive_name>.tar.gz
├── manifest.yaml          ← always at archive root
├── etc/hosts              ← files stored with absolute path, leading / stripped
├── etc/nginx/nginx.conf
└── home/jerry/.bashrc
```

Files are stored with paths relative to `/` (i.e. `etc/hosts`, not `/etc/hosts`).
Extract with: `sudo tar -xzf <archive> -C /`

## manifest.yaml Schema example

```yaml
porter_manifest:
  version: "1.0"
  created: "2026-03-14T15:23:01"     # ISO 8601, seconds precision
  hostname: "homelab-node1"
  source_base: "/"                    # base directory snapshotted — "/" for system snapshots
  file_count: 42                      # number of entries in the files list

os:
  pretty_name: "Ubuntu 24.04 LTS"    # from /etc/os-release PRETTY_NAME
  id: "ubuntu"                        # from /etc/os-release ID
  version_id: "24.04"                 # from /etc/os-release VERSION_ID
  kernel: "6.8.0-51-generic"          # platform.release()
  arch: "x86_64"                      # platform.machine()

extraction:
  command: "sudo tar -xzf <archive_name> -C /"
  notes:
    - "Files are stored with absolute paths from /."
    - "Extract to / to place files in their correct locations."
    - "Verify uid/gid match after extraction — create users first if needed."
    - "Enable any required systemd services after extraction."

local_users:                          # non-system users (uid 1000–60000)
  - username: "jerry"
    uid: 1000
    gid: 1000
    primary_group: "jerry"
    home: "/home/jerry"
    shell: "/bin/bash"

systemd_services:
  note: "Services that were active on the source system"
  active:
    - "nginx.service"
    - "postgresql.service"
    - "ssh.service"

packages:
  manager: "apt/dpkg"                 # or "rpm" or "pacman" or "unknown"
  count: 847
  installed:
    - name: "nginx"
      version: "1.24.0-2ubuntu7"
    - name: "postgresql-16"
      version: "16.3-1.pgdg24.04+1"
    # ... full package list from dpkg/rpm/pacman

files:                                # one entry per changed/new file
  - path: "/etc/hosts"               # absolute path on source system
    archive_path: "etc/hosts"        # path as stored in the archive (no leading /)
    status: "MOD"                    # "MOD" = modified, "NEW" = new file
    mode: "0o644"                    # octal permission string
    owner: "root"
    group: "root"
    uid: 0
    gid: 0
    size_bytes: 312
    modified: "2026-03-14T14:55:00"  # ISO 8601, seconds precision
    sha256: "a3f1..."                # SHA-256 hex digest of file contents at snapshot time
```

## Labinator Integration Notes

### What labinator should do with this manifest

1. **Read `porter_manifest.source_base`** — if it's `/`, this is a full system diff archive;
   if it's a subdirectory, it's a targeted app config diff.

2. **Check `os`** — Warn if the source OS doesn't match the target container's OS.

3. **Pre-extraction: create users** — iterate `local_users`, create any that don't exist on
   the target before extracting. The `uid`/`gid` values must match because tar preserves
   numeric ownership and system users (uid < 1000) are expected to already exist from
   package installs.

4. **Verify packages** — `packages.installed` is a snapshot of what was installed
   on the source. Labinator could cross-reference this against `apt list --installed` on the
   target and warn about missing packages.

5. **Extract the archive** — `sudo tar -xzf <archive> -C /`. The `extraction.command` field
   documents this explicitly.

6. **Post-extraction: enable services** — iterate `systemd_services.active`, enable and start
   any that aren't already running: `systemctl enable --now <service>`.

7. **Verify file integrity** — `files[*].sha256` lets labinator confirm extracted files
   match the source. Useful for post-deploy validation.

### Key caveats

- **`status` field** — only `"MOD"` and `"NEW"` appear. Deleted files are not tracked (porter
  snapshots what exists, not what was removed).
- **Symlinks excluded** — the diff walk uses `followlinks=False` and only records regular
  files (`os.walk` filenames). Symlinks and directories are not listed in `files`.
- **Hidden files excluded from diff walk** — directories starting with `.` are pruned during
  the walk (`not d.startswith(".")`). Files starting with `.` within non-hidden directories
  are included.
- **uid/gid caveat** — numbers must match on the destination. The recommended labinator
  workflow: install packages first (so system users exist), create `local_users` next,
  then extract the archive.
- **`packages` is informational** — it lists what was installed on the source, not what the
  archive requires. Labinator uses it as a reference, not a dependency resolver.

## Source Reference

Manifest is built by `_build_manifest()` in `~/projects/porter/porter/app.py`.
Archive is built by `_build_archive()` (background thread) in the same file,
triggered from `action_build_archive_from_diff()`.
